准 L3+ 智驾的风险评价模型训练体系
是什么 · 怎么建 · 嵌哪里

回应集团提出的命题：「针对智驾（非传统 ADAS、准 L3 以上）的风险评价模型训练体系，希望有一个输出，以便尽快推动嵌入主机厂智驾服务体系。」本专题给出可讨论、可启动的完整框架 — 从今天就能确认的数据清单开始。
为什么传统 UBI 不够用

同一辆车、同一条路：智驾激活时的急加速是系统标定问题（按版本归因），人驾时是驾驶习惯问题（按人归因）；而智驾相关事故高度集中在接管前后的几秒钟* — 必须把人驾、系统驾、切换过程分成三个通道分别建模。

* 行业研究典型规律，量级示意；以试点实测为准

先看一张图：训练体系到底是一条什么「生产线」

一句话：把智驾风险变得「可观测 → 可归因 → 可定价」的数据生产线，并且每次 OTA 后自动重新学习
它不是一个一次性建好的模型，而是一条持续运转的生产线：左边进智驾数据，右边出费率因子、发版风险评估和用户智驾分 — 软件每升一版，体系自动重评一次。
投料
① 数据盘点与接入 按 A/B/C 三级可得性签数据清单，与 UBI 共用接入管线
加工
② 场景库与责任标注 事件切片入库，按「人 / 系统 / 混合」三类归因标注
核心
③ 三通道风险模型 人驾 / 系统驾 / 切换分别建模 — 风险跟着「谁在开」走
质检
④ 影子验证与回归 新版本先并行打分不生效，与赔案回测后才进费率
出货
⑤ 嵌入智驾服务体系 费率因子 · OTA 发版评估 · 用户智驾分 · 责任判定依据
闭环是关键：OTA 发版 → 新数据回流 → 自动重训重评。智驾能力随软件版本跳变，「训练体系」的本质是持续重学的机制，而不是某一个模型本身 — 这正是它必须嵌入主机厂智驾服务体系、而不能放在体系外的原因。

车端信号的三级可得性 — 决定起步速度

不要等「全部数据」：A 级签清单就能动，B/C 级随合作深化逐级解锁
A
整车 CAN 即可得明天就能谈
智驾功能激活/退出状态 · 车速与驾驶员操作 · AEB/FCW 触发事件 · 里程与时段 — 多数信号已在 UBI 接入管线口径内
解锁：通道 A 人驾风险 + 智驾暴露度统计
B
智驾域控日志需接口协作
接管请求 TOR 时序 · ODD 进出事件 · MRC 最小风险策略触发 · 脱手/分神监测（DMS）— L3 准入法规本就要求记录
解锁：通道 C 切换风险 + 版本×场景矩阵骨架
C
研发深度协作联合建模
感知置信度与传感器降级 · 事件前后 30s 场景切片 · EDR / 影像数据 — 与智驾研发的数采-标注体系共用基建
解锁：通道 B 系统驾风险全量 + 责任判定依据
给管理层的判断：起步不需要新开发车端能力 — A 级信号现在就有；B 级是 L3 法规的既有要求；只有 C 级需要研发侧投入。数据清单可以按 A→B→C 分三次签。

问一训练体系是什么 — 三通道风险结构

同一段行程按「谁在开」切分为三个风险面，分别建模、统一输出到 200–997 同尺度评分　　* 「在环」即 in-the-loop：指此刻由谁闭合「感知→决策→操作」的驾驶控制回路，风险就归因到谁
通道 A · 人驾风险
人在开 驾驶员在环 *
UBI 从车评分的自然延伸：智驾未激活里程沿用 4 项指标体系，口径与现有平台一致、模型资产直接复用。
里程暴露急加速/急转快充/温度时段与路线
输出 → 人驾风险分（与现行从车评分同尺度 200–997）
通道 B · 系统驾风险
系统在开 智驾系统在环 *
风险归因到「软件版本 × ODD 场景」而非个人：同一版本在同类场景的群体表现，才是系统风险的正确统计单元。
版本号分层ODD 内/外MPI 接管间隔MRC 触发率传感器置信度
输出 → 版本×场景风险矩阵（每次 OTA 后重新评估）
通道 C · 切换风险
正在交接 人机接管过程
接管请求到稳定控制的全过程：响应时延、接管时车速与车距余量、接管后 10 秒内的修正幅度 — 事故最密集的窗口。
接管响应时延脱手/分神时长接管后修正幅度ODD 边界预判
输出 → 接管质量分（人与系统配合度，双向归因）
一段行程的责任切分 风险通道随驾驶状态机切换，秒级对齐遥测时间轴

问二训练体系怎么建 — 五阶段流水线 + 四条铁律

数据 → 场景 → 模型 → 验证 → 嵌入，与主机厂智驾研发流程（数采-标注-回归）共用基建，不另起炉灶
阶段 ①
数据采集与对齐
  • 按 A/B/C 清单分批接入（见上文盘点）
  • 智驾状态机：激活/退出/接管请求/MRC 全事件流
  • 与 UBI 遥测共用接入管线，秒级时间对齐
in: CAN/智驾域控日志
out: 统一时序事件流
阶段 ②
场景库与责任标注
  • 事件前后 30s 切片，按场景分类（匝道/施工/cut-in…）
  • 责任标注：人 / 系统 / 混合 三类归因
  • corner case 入库，持续扩充对抗集
in: 时序事件流
out: 带标注场景库
阶段 ③
三通道模型训练
  • 通道 A：沿用 UBI 因子体系（迁移，不重建）
  • 通道 B：版本 × ODD × 场景分层回归
  • 通道 C：接管过程序列模型（时延/余量/修正）
in: 场景库 + 标注
out: 三通道风险模型
阶段 ④
评价与验证
  • 影子模式：新版本先并行打分不生效
  • OTA 回归：版本对比 + PSI 漂移监控
  • 与已决赔案回测，赔付率区分度验收
in: 模型 + 新版本数据
out: 版本风险报告
阶段 ⑤
嵌入智驾服务体系
  • 智驾保险费率因子（按激活里程/ODD 计价）
  • OTA 发版风险评估前置
  • 用户智驾分进入车主 App
in: 三通道评分
out: 费率/发版/运营接口
版
按软件版本分层，OTA 即重评智驾能力随 OTA 跳变，跨版本聚合会污染统计；每次发版触发影子评估。
域
ODD 内外严格区分ODD 内看系统、ODD 外看人、边界看切换；混在一起的「智驾出险率」没有定价意义。
序
时序行为优先于静态标签接管质量看连续过程，而非「是否发生接管」的一次性标记。
基
与 UBI 平台共用数据底座接入、清洗、合规域内计算全部复用 P1 基建 — 智驾模型是扩展，不是另起炉灶。

问三嵌入智驾服务体系哪里 — 四个挂点 + 固定交付物

不改变主机厂现有研发与服务流程，以接口方式挂载；每个挂点对应一类固定交付物
嵌入一汽智驾服务体系的四个挂点 体系建在保险侧，挂点接在主机厂侧 — 双向受益、互不侵入
研发侧
数采与回归平台场景库与责任标注与智驾研发共用，高频接管点位 = 算法迭代优先级清单
发版侧
OTA 发布流程风险影子评估作为发版检查项之一，发版后 30 天追踪
服务侧
智驾订阅与保险费率因子接入智驾订阅 / 智驾保险产品定价（按激活里程 / ODD 计价）
用户侧
车主 App用户智驾分与 ODD 透明化 — 让用户知道何时可用、用得多好，建立信任

固定交付物

SDI

200–997
三通道风险评分人驾 / 系统驾 / 切换，与从车评分同尺度
每版本
OTA 风险回归报告发版前影子评估 + 发版后 30 天追踪
API
智驾保险费率因子按激活里程 / ODD 场景计价的精算接口
从明天开始的 90 天 不等全部数据、不等法规完备 — 用 A 级信号先把生产线跑起来。数据准备与 P1 并行预埋，费率应用待 P3 窗口正式启用
第 1–30 天
签 A 级数据清单与一汽智驾部门确认司南功能阶梯与 A 级信号口径，纳入既有 UBI 合规框架
第 30–60 天
影子接入跑通智驾激活状态 + 暴露度统计上线从车风险数据平台「智驾风险」模块
第 60–90 天
首版基线报告人驾 vs 智驾暴露与事件率基线 + B/C 级数据解锁路线图，提交双方管理层
